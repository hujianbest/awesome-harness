# AHE 收尾

把当前 AHE 工作周期干净地收尾。

## 硬性门禁

只有在 `ahe-completion-gate` 已通过后，才能进入本 skill。

如 `AGENTS.md` 为当前项目声明了完成门禁记录的等价路径、状态别名或证据位置，按其中映射读取；但这不会替代 `ahe-completion-gate` 本身。

如果完成门禁状态不清楚、证据仍不新鲜，或当前工作仍有未处理阻塞，应先回到 `ahe-workflow-starter` 重新编排，而不是提前收尾。

收尾阶段不再混入新的实现、修复、设计或评审工作。

## 目的

这个 skill 的目标是把“已经完成的工作”沉淀为“可持续的项目状态”。

它的存在是为了让下一次会话不必重新摸索：

- 做了什么
- 有哪些证据支持完成
- 对用户来说有哪些变化
- 接下来应该做什么

## 前置条件

只有在当前任务通过完成门禁之后，才能使用这个 skill。

默认应结合以下状态工件一起完成收尾：

- `task-progress.md`
- `RELEASE_NOTES.md`
- `docs/reviews/`
- `docs/verification/`

## Integration

通常由 `ahe-workflow-starter` 在确认当前 profile 已完成实现链路、且 `ahe-completion-gate` 为 `通过` 后把会话带到这里。

本 skill 主要读取：

- `task-progress.md`
- `RELEASE_NOTES.md`
- 当前轮相关的 review / verification 记录
- `AGENTS.md` 中与收尾、发布说明、状态记录相关的映射和覆盖约定

本 skill 主要写回：

- `task-progress.md` 中的阶段、下一步和 session 记录
- `RELEASE_NOTES.md` 中与本轮相关的用户可见变化
- 本轮完成证据的索引或引用位置

收尾完成后，如果工作流没有结束、后续还会继续推进新的任务或支线，下一次恢复仍应从 `ahe-workflow-starter` 开始判断，而不是默认沿用聊天记忆。

## 可重复执行性

`ahe-finalize` 应被视为可重复执行的收口动作。

如果因为遗漏记录、补写发布说明或同步额外证据而需要再次进入本 skill：

- 优先更新现有状态和记录
- 不重复制造相互矛盾的完成结论
- 不引用旧的、过期的门禁或验证结果替代当前证据

## 红旗信号

出现以下想法时，先停下。这通常说明你正在为“跳过收尾”找理由：

| 想法 | 实际要求 |
|---|---|
| “已经做完了，不用再写记录” | 收尾的目标就是把完成状态沉淀为可继续的项目状态。 |
| “这些证据我自己知道在哪，不用写出来” | 证据位置必须显式记录，便于后续追溯。 |
| “下一步别人自己看代码就知道了” | 交接信息必须明确说明下一步和剩余工作。 |
| “发布说明以后再补也没关系” | 只要影响用户可见行为，就应在收尾时同步记录。 |
| “顺手再改一点实现再一起收尾” | 收尾阶段不再混入新的实现工作。 |
| “状态先不更新，后面想起来再说” | 进度状态、同步状态和交接信息必须在当前收尾中完成。 |

## 工作流

### 1. 更新进度状态

在项目进度记录中补充：

- 已完成任务或范围项
- 日期或会话标记
- 当前状态
- 推荐的下一任务
- 当前阶段
- 下一步动作或推荐 skill
- 已批准工件摘要

如果当前主链任务全部完成，应在状态记录中明确标记工作流完成，而不是只在对话中说“结束了”。

### 2. 更新发布说明

如果完成的工作影响了用户可见行为，请在发布说明中简要记录：

- 变了什么
- 为什么重要

### 3. 记录证据位置

记录支持完成结论的证据在哪里：

- 缺陷模式排查结果（如有）
- 测试执行结果
- 评审结论
- 追溯性评审结果
- 回归结果
- 完成门禁结果

记录可以简洁，但必须支持后续审计和追溯。

完成门禁和本轮相关的 review / verification 结果应已按 workflow 要求落盘。这里必须优先记录这些仓库内文件路径，而不是只写口头摘要。

如 `AGENTS.md` 为当前项目声明了等价路径，按其中映射路径记录。

引用证据时，优先使用当前最新一次与本轮工作直接相关的结果；不要把旧的通过记录当作本轮 fresh evidence。

### 4. 准备交接

明确说明：

- 哪些内容已经完成
- 还剩什么
- 哪些受影响工件已经同步刷新
- 如果下一次会话继续，推荐的下一步动作或 skill 是什么

### 5. 维护状态收口

确保以下内容已同步：

- `task-progress.md` 中的 Current Stage
- `task-progress.md` 中的 Current Active Task
- `task-progress.md` 中的 Next Action Or Recommended Skill
- `task-progress.md` 中的 Session Log
- 与本轮相关的证据索引

如果这些内容尚未全部同步完成，就不要把收尾说成已经完成。

## 输出格式

请严格使用以下结构：

```markdown
## 已完成工作

- 已完成项

## 已更新记录

- 已更新记录项

## 证据摘要

- 证据项

## 证据位置

- 路径

## 同步状态

- 已同步工件

## 下一步

`回到 ahe-workflow-starter 继续编排` | `回到 ahe-workflow-starter 进入变更处理` | `回到 ahe-workflow-starter 进入热修复处理` | `工作流完成`
```

## 反模式

- 宣告完成，却没有更新项目状态
- 把下一任务留成隐含信息
- 对用户可见变化忘记写发布说明
- 明明发生过变更或热修复，却不确认受影响工件是否已同步
- 在收尾过程中混入新的实现工作
- 没有记录证据位置，就直接结束当前会话
- 没有更新 `task-progress.md` 中的阶段和下一步，却声称已经完成收尾

## 完成条件

只有在项目状态、发布沟通和交接信息都更新到足以让下一次会话无须猜测地继续时，这个 skill 才算完成。
