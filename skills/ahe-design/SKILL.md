# AHE 设计

创建设计文档，说明已批准规格将“如何”被实现。

## 硬性门禁

在设计经过评审并获批之前，不得拆解任务或编写实现代码。

在 `ahe-design-review` 给出“通过”结论之前，不发起真人批准确认。

如果当前请求尚未经过 `ahe-workflow-starter` 的阶段判断，不直接开始设计工作；应先回到 `ahe-workflow-starter` 完成路由。

## 前置条件

仅在以下条件满足时使用本 skill：

- 已存在需求规格
- 该规格已被明确批准

如果规格仍是草稿、评审虽通过但尚未完成真人确认，或仍在修订中，请回到 `ahe-spec-review` 或 `ahe-specify`。

## 适用时机

通常在以下场景使用本 skill：

- 需求规格已经批准，下一步需要明确 HOW
- 当前工作在进入任务规划前，需要先确定模块、接口、数据流和主要取舍
- 设计评审退回后，需要基于评审意见继续收敛设计

## 工件与状态要求

默认情况下，设计文档应保存到：

- `docs/designs/YYYY-MM-DD-<topic>-design.md`

若 `AGENTS.md` 为当前项目声明了设计路径映射，优先使用映射路径；否则使用默认路径。

在设计草稿交评审前，文档中至少应体现：

- 状态字段，例如 `状态: 草稿`
- 主题或范围标识
- 可供评审定位的章节结构

若项目尚未形成固定进度记录格式，默认使用：

- `templates/task-progress-template.md`

设计草稿交给 `ahe-design-review` 后，后续真人确认完成时还应同步：

- 设计文档状态
- `task-progress.md` 中的 Current Stage
- `task-progress.md` 中的 Next Action Or Recommended Skill
- review 记录中的 Human Confirmation 与 Evidence Location

## 目标

设计文档应在保持设计层抽象的前提下，尽量消除后续实现中的主要猜测空间。

它应回答以下问题：

- 存在哪些主要组件或模块
- 职责如何划分
- 数据和控制流如何在系统中流动
- 需要哪些接口
- 哪些约束会影响实现
- 设计如何支撑测试

## 红旗信号

出现以下想法时，先停下。这通常说明你正在为“跳过设计推敲”找理由：

| 想法 | 实际要求 |
|---|---|
| “规格已经有了，直接拆任务就行” | 设计未评审前，不进入任务拆解。 |
| “这个方案看起来最顺手，不用比较备选方案” | 至少比较两个可行方案后再定稿。 |
| “先把实现细节写进去，后面省事” | 设计要保持设计层抽象，不写成逐行实现说明。 |
| “非功能需求影响不大，可以先忽略” | 约束和非功能需求必须体现在设计决策里。 |
| “接口先模糊写一下，后面再补” | 关键接口与契约要足够明确，才能支撑任务规划。 |
| “测试以后再考虑” | 设计阶段就要说明测试策略。 |

## 工作流

### 1. 阅读已批准规格

提取：

- 核心范围
- 验收标准
- 约束
- 非功能需求
- 集成点

优先读取：

- `AGENTS.md` 中与设计相关的路径映射、模板覆盖和团队设计约束
- `task-progress.md` 中的当前阶段记录（如果存在）

### 2. 了解技术上下文

阅读最少但必要的技术上下文：

- 现有架构或项目布局
- 当前框架和依赖
- 已知的部署或运行约束

此时不要提前进入实现规划。

### 3. 提出 2 到 3 个候选方案

在最终选定设计前，至少比较两个可行方案。

对每个方案，简要说明：

- 它如何工作
- 主要优点
- 主要缺点
- 对关键约束、非功能需求和集成点的影响
- 为什么适合或不适合当前规格

如果两个方案都无法满足关键约束或主要非功能要求，不要勉强推荐；应先显式指出当前设计仍存在阻塞点。

若存在满足关键约束的可行方案，最后推荐一个方案；否则标记当前设计仍需继续澄清或修订，不进入评审。

如果当前是因为 `ahe-design-review` 返回“需修改”或“阻塞”而重新进入本 skill：

- 先读取评审结论中的发现项、薄弱点和修订建议
- 优先修复阻塞任务规划的问题
- 只针对仍然影响设计定稿的问题向用户补充确认
- 不要把已确认且未受影响的设计内容全部重做

### 4. 编写设计文档

若 `AGENTS.md` 为当前项目声明了设计模板、章节骨架或记录要求，优先遵循这些约定。

若 `AGENTS.md` 未提供设计模板覆盖，则使用以下最小结构：

```markdown
# <主题> 实现设计
## 概述
## 设计驱动因素
## 候选方案
## 选定方案
## 架构设计
## 模块职责
## 数据流与控制流
## 接口与契约
## 测试策略
## 风险与待定问题
```

默认应先遵循 `AGENTS.md` 中的路径映射与模板覆盖；若未提供，再使用本 skill 的默认路径和上述章节骨架。

编写建议：

- 区分需求与设计
- 清晰定义边界和职责
- 解释为什么所选方案适合该规格
- 提供足够支撑任务拆解的细节，但不要写成逐行编码说明
- 若仍存在不阻塞评审的待确认点，在设计文档中显式留下开放问题，而不是假装全部已定

### 5. 评审前自检

确认以下内容：

- 每个主要需求都被设计覆盖
- 约束和非功能需求确实影响了设计决策
- 架构是连贯的，而不是组件清单
- 接口定义足够明确，能支撑后续任务规划
- 存在设计层面的测试策略
- 测试策略说明的是测试准备度、风险覆盖思路和分层方式，而不是任务级测试细案

### 6. 交给评审

只有在存在可行推荐方案、且设计草稿已经达到可评审状态时，才交给 `ahe-design-review`。

不要在 `ahe-design` 阶段请求真人批准设计；真人确认只发生在 `ahe-design-review` 给出“通过”之后。

若项目尚未形成固定 review 记录格式，默认由 `ahe-design-review` 使用：

- `templates/review-record-template.md`

交接前请同步：

- 将设计草稿保存到约定路径
- 在 `task-progress.md` 中将 Current Stage 更新为设计评审前状态
- 在 `task-progress.md` 中将 Next Action Or Recommended Skill 更新为 `ahe-design-review`

输出应使用：

```markdown
设计文档草稿已起草完成，请提交 `ahe-design-review` 进行评审。

推荐下一步动作或 skill: `ahe-design-review`
```

如果这是基于评审意见修订后的版本，也使用同样交接方式重新进入 `ahe-design-review`。

如果当前仍存在阻塞性设计问题，输出应改为：

```markdown
当前设计仍存在阻塞性问题，先继续澄清或修订后再进入 `ahe-design-review`。

推荐下一步动作或 skill: `ahe-design`
```

注意：只有 `ahe-design-review` 返回“通过”后，才进入真人确认；真人确认通过后才能进入 `ahe-tasks`。

## 反模式

- 把设计文档写成实现伪代码
- 只是复制需求规格，没有加入设计决策
- 只给一个方案，不讨论权衡
- 在设计文档里直接开始拆任务
- 忽略非功能需求和约束

## 完成条件

只有在以下两种情况之一满足时，这个 skill 才算完成：

- 已形成一份可评审的设计文档，并且已准备好交给 `ahe-design-review`
- 已明确指出当前设计仍存在阻塞性问题，且给出了继续留在 `ahe-design` 修订所需的下一步
