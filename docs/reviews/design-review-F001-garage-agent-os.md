# 设计评审记录：F001 Garage Agent 操作系统

- **评审对象**: `docs/designs/2026-04-15-garage-agent-os-design.md`
- **关联规格**: `docs/features/F001-garage-agent-operating-system.md`
- **评审日期**: 2026-04-15
- **评审角色**: 独立 reviewer subagent
- **结论**: **需修改**

---

## 结论

**需修改**。设计文档整体质量较高，架构分层清晰，ADR 记录完整，需求追溯矩阵覆盖了绝大多数功能需求。但存在若干可操作的发现项，核心关注点集中在：(1) 关键数据结构缺少版本化 schema 定义；(2) artifact-first 与 board-assisted 双源一致性缺乏显式冲突检测协议；(3) NFR 验证方法缺乏具体量化度量手段；(4) 多处接口定义使用了伪类型语法而非可机器校验的 schema。这些缺口不阻塞架构方向，但在进入 ahe-tasks 前应补齐。

## 发现项

### F-01: Artifact-Board 双源冲突缺乏显式检测协议

- **严重度**: High
- **描述**: ADR-003 声明"若 board 与工件冲突，以工件为准"，但设计未定义：(a) 如何检测冲突（什么粒度比较？逐字段还是整体 hash？）；(b) 冲突发生时的具体处理步骤（是覆盖 board？还是标记冲突让用户决定？）；(c) 冲突日志记录格式。
- **影响**: 在实际运行中，artifact 被 Claude Code 修改（用户手动编辑）后，board 可能持有过时信息，导致 session 恢复时进入不一致状态。
- **建议**: 在第 9 章增加一节"Artifact-Board 一致性协议"，定义：检测时机（session 恢复时、skill 执行前）、比较方法（front matter hash 或关键字段）、冲突解决规则、日志格式。
- **工作量**: 约 1-2 小时（文档补充）
- **优先级**: P1 — 进入任务拆解前补齐

### F-02: 接口契约使用非标准伪语法

- **严重度**: Medium
- **描述**: 第 10 章的接口契约使用自定义 YAML 伪语法（如 `interface HostAdapter:` + `method invoke_skill:`），这不是 JSON Schema、OpenAPI、或任何可机器校验的标准格式。对于一个"文件即契约"的系统，契约文件本身应该使用可校验的 schema 格式。
- **影响**: 无法自动化校验接口合规性；实现者对接口的理解可能产生歧义。
- **建议**: (1) 明确声明当前为 Phase 1 的"人类可读契约"，Phase 2 迁移到 JSON Schema；(2) 在 contracts/ 目录下放置一个 README.md 说明契约格式约定；或 (3) 直接使用 JSON Schema 重新定义核心接口。
- **工作量**: 约 2-3 小时
- **优先级**: P2 — 可在任务拆解后、实现前补齐

### F-03: NFR 验证方法缺乏量化度量协议

- **严重度**: Medium
- **描述**: 第 11 章 NFR 检查表的"验证方法"列几乎全是定性描述（如"性能测试""安全审计""代码审查"），缺少：(a) 具体的度量方法（如何测量 p90 响应时间？）；(b) 测试工具选择；(c) 通过/失败的具体阈值；(d) 退化基线的建立时机。
- **影响**: 进入实现阶段后，开发者无法判断系统是否满足 NFR，导致 NFR 形同虚设。
- **建议**: 为每个 NFR 补充一条具体的验证协议。例如："NFR-001 性能：在知识库包含 100 条目时，运行 `measure_skill_response.py` 脚本执行 100 次知识查询，计算 p90 响应时间，目标 < 5s。"
- **工作量**: 约 2-3 小时
- **优先级**: P2 — 在实现对应模块前补齐

### F-04: Session 恢复中 checkpoint 链损坏的极端场景未覆盖

- **严重度**: Medium
- **描述**: 第 9.3 节描述了 session 恢复流程，包括"如果损坏，尝试从 checkpoints 恢复"。但未考虑：(a) checkpoint 本身损坏的情况；(b) 所有 checkpoint 都损坏的情况；(c) checkpoint 与 session.json 都损坏但 artifacts 仍存在的情况。
- **影响**: 在极端情况下（磁盘部分损坏），系统可能无法恢复且无明确的降级策略。
- **建议**: 补充一个 checkpoint 损坏的降级策略：校验 checkpoint 完整性 → 回退到上一个有效 checkpoint → 若全部损坏，提供"从头开始但复用已有 artifacts"的选项。
- **工作量**: 约 1 小时
- **优先级**: P2

### F-05: 隐藏假设 — Claude Code session 状态可被外部持久化

- **严重度**: Medium
- **描述**: 设计假设可以通过 Host Adapter 读取和写入 Claude Code 的 session 状态（FR-001b、FR-003），但 Claude Code 的 session API 能力是有限的。设计未记录这个假设的失效风险和缓解措施（规格的 ASM-001 提到了但设计未承接）。
- **影响**: 如果 Claude Code 不支持所需的 session 操作，整个状态管理设计需要调整。
- **建议**: 在第 16 章待定问题中增加一条："验证 Claude Code 的 session API 是否支持设计所需的操作（读取状态、写入检查点、恢复到指定状态）"，并在 Phase 1 第一个任务中安排一个技术验证 spike。
- **工作量**: 约 0.5 小时（文档补充）+ 技术验证 spike
- **优先级**: P1 — 应在任务拆解中安排为第一个任务

### F-06: 知识文件格式缺少版本化 schema 声明

- **严重度**: Medium
- **描述**: 第 10 章定义了 knowledge 和 experience 的文件格式示例，但未包含 `schema_version` 字段。设计原则"文件即契约"要求所有 schema 变更必须有版本号，但实际文件模板中没有版本标识。
- **影响**: 未来 schema 变更时无法区分新旧格式，违反"文件即契约"原则中的版本管理要求。
- **建议**: 在所有数据文件的 front matter / JSON 中增加 `schema_version: "1"` 字段，并在 platform.json 中声明当前支持的 schema 版本范围。
- **工作量**: 约 1 小时
- **优先级**: P1 — 直接影响数据格式设计

### F-07: FR-007 工具扩展接口缺少具体调用时序

- **严重度**: Low
- **描述**: 工具注册表定义了数据结构（registered-tools.yaml），Tool Gateway 作为模块存在，但缺少工具调用的具体时序图。FR-007 的验收标准提到"工具链编排和数据传递"，但设计中无对应的数据流。
- **影响**: 实现工具扩展功能时缺少明确的执行路径指导。
- **建议**: 在第 9 章增加一个"工具调用流程"的时序描述。鉴于 FR-007 是 Should 优先级且 Phase 1 可以简化实现，此发现项优先级较低。
- **工作量**: 约 1 小时
- **优先级**: P3

## 薄弱/缺失设计点

1. **AGENTS.md 约定中心内容不完整**: 当前 AGENTS.md 只有 3 条约定（规格路径、项目灵魂、术语说明），缺少设计文档中提到的路径映射、审批等价证据、禁并发规则、模板覆盖等约定。设计文档引用了这些约定但实际的 AGENTS.md 并未包含。建议在实现前补齐 AGENTS.md。

2. **contracts/ 目录实际为空**: 设计文档定义了 contracts/ 下的多个 YAML 文件（workflow-session.yaml、node-definition.yaml 等），但这些文件当前可能不存在。建议在任务拆解中将"创建 contracts/ 目录和初始契约文件"列为前置任务。

3. **成长策略与设计阶段的映射**: 设计提到了 Stage 1-4 但未显式引用 `docs/soul/growth-strategy.md` 的内容。建议在设计文档中增加一节说明当前设计与成长策略各阶段的对应关系。

4. **错误处理中的"Ignorable"分类**: 第 12.2 节错误处理四层次包含"Ignorable"类型，但错误分类 schema（第 8.3 节 Error Handler）中只有 retryable、user_intervention、fatal 三类，缺少 ignorable。两处不一致。
