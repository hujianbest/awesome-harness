# Skill Reviewer

评审一个 skill 的整体质量，判断它是否已经足够清晰、可触发、可验证、可维护，并区分“可以继续迭代”与“已经适合共享或发布”。

## 硬性边界

这个 skill 默认做审查，不负责直接改写 skill。

如果用户明确要求修订、重写或继续补齐，可以先完成本轮审查，再把发现项交给后续修订流程；不要把审查结果悄悄替换成重写稿。

如果用户只给了部分文件，只对已见范围下结论，不要假装已经完成整包审查。

## 目标

一次合格的 skill 质量审查，至少要回答这些问题：

- 这个 skill 会不会在合适场景被触发
- `SKILL.md` 本体是否清晰、自洽、边界稳定
- 包结构是否支持使用者理解和维护
- `evals` 是否真的在验证该 skill，而不是自说自话
- 是否存在让用户意图产生“惊喜”的隐藏行为
- 如果用户关心共享或发布，这个 skill 还缺哪些治理项

## 快速工作模式

| 情况 | 当前回合应做什么 | 不要做 |
|---|---|---|
| 用户只给了 `SKILL.md` | 只审主文件质量，并明确哪些包级维度尚未评估 | 不要把单文件审查看成整包审查 |
| 用户要求“审整个 skill 包” | 至少检查 `SKILL.md`，再按需看 `README`、`evals`、`references`、`scripts` | 不要只看 README 或只看 frontmatter |
| skill 没有 `scripts` 或 `references` | 先判断这类资产是否真的必要 | 不要机械地把“没有脚本”当成缺陷 |
| 用户问“能不能共享或发布” | 区分核心质量问题与治理、分发准备度问题 | 不要把许可证、版本说明缺失一律判成实现阻塞 |
| 用户要你直接改写 skill | 先给审查结论，再说明是否进入修订 | 不要跳过 review 直接重写 |
| `evals` 已存在 | 检查 prompt、期望和正文规则是否对齐 | 不要只因为有 `evals.json` 就默认评测准备度合格 |

## 工作方式

### 1. 先确认审查对象和范围

优先弄清以下信息：

- 当前要审的是单个 `SKILL.md`，还是整个 skill 包
- 用户是关心“可继续迭代”，还是关心“适合共享或发布”
- 用户是否希望你做静态审查，还是顺便做一轮轻量验证
- 当前哪些文件真实可读，哪些只是用户口头提到

如果用户没有明确说明，默认按“静态审查 + 只对已见范围负责”处理。

### 2. 读取最小必要输入

最少读取：

- `SKILL.md`

如果用户要求整包审查，继续按需读取：

- `README.md`
- `evals/README.md`
- `evals/evals.json`
- `references/` 中被 `SKILL.md` 明确依赖的文件
- `scripts/` 中被正文要求执行或引用的脚本

如果 skill 声称依赖某个文件而该文件不存在，或引用路径明显失效，要把它记为审查发现项。

### 3. 先判断 skill 类型和预期资产

先判断当前 skill 更接近哪类：

- 轻量自包含型：主要靠 `SKILL.md` 就能完成
- 文档型：需要 `README` 或 `references`
- 工具型：依赖 `scripts/`
- 评测驱动型：依赖 `evals`

不同类型对“完整包”的要求不同。

例如：

- 一个纯思维或流程型 skill 可能不需要脚本
- 一个复杂 reference skill 没有补充文档就可能不够完整
- 一个声称“可稳定复用”的 skill，如果完全没有评测样例，共享准备度通常会偏弱

### 4. 按维度审查

#### 4.1 触发与描述

检查：

- `name` 是否清晰、稳定、便于搜索
- `description` 是否主要描述“何时使用”，而不是把正文流程压缩进 frontmatter
- 触发词是否覆盖用户真实会说的话
- 描述是否过宽，导致误触发
- 描述是否过窄，导致漏触发
- 描述与正文范围是否一致

#### 4.2 核心说明质量

检查 `SKILL.md` 是否：

- 讲清了目标、边界和不适用场景
- 有稳定的工作顺序，而不是零散建议
- 能区分必须做、可选做和不该做
- 输入、输出、停机条件是否明确
- 是否存在自相矛盾、重复规定或隐含规则

#### 4.3 包结构与渐进披露

检查：

- `README` 是否补充使用方式，而不是机械重复正文
- 详细参考是否被合理拆到独立文件
- 正文引用的文件是否真实存在、命名清楚、层级不过深
- 包结构是否与 skill 的复杂度匹配
- 是否把大量低频细节硬塞进 `SKILL.md` 导致主体失焦

#### 4.4 评测准备度与对齐性

如果存在 `evals`，重点检查：

- 评测 prompt 是否像真实用户请求，而不是抽象口号
- 是否覆盖正向场景、边界场景、近似误用场景
- `expected_output` 或 `expectations` 是否真的在验证正文规则
- 评测是否奖励了与正文相冲突的行为
- audit-only skill 的评测，是否错误要求它直接改写内容
- 如果缺少 `evals`，判断这是轻微缺口、重要缺口，还是在当前 skill 类型下可以接受

#### 4.5 一致性与可维护性

检查：

- 术语是否统一
- 默认路径、用户覆盖路径、模板覆盖规则是否讲清
- README、SKILL、evals 之间是否互相打架
- 是否存在一处改了、另一处没同步的漂移迹象
- 规则是否足够稳定，后续维护者能否低成本更新

#### 4.6 安全与意外性

检查：

- 是否要求做出超出用户意图的动作
- 是否暗示危险、误导、越权、数据外泄或破坏性操作
- 是否把“审查”偷偷变成“自动改写”或“自动发布”
- 是否会让使用者对 skill 的行为产生不必要惊喜

#### 4.7 共享与发布准备度

只有在用户关心共享、发布、对外分发时，才重点放大这一维。

检查：

- 是否有足够的 `README` 帮助别人理解和调用
- 是否需要许可证、版本说明、变更记录或兼容性说明
- 是否存在明显依赖仓库内隐式上下文，导致离开当前仓库就难以复用
- 缺少这些治理项时，要明确它们属于“共享准备度缺口”，不必一律上升为核心阻塞

### 5. 如有需要，补一轮轻量验证

如果用户明确关心“可发布”“可安装”“可交付”，并且仓库里已有现成校验脚本，可以补一轮轻量验证，例如：

- frontmatter 格式校验
- 被正文引用的关键文件是否存在
- `evals` 数据结构是否至少可读
- skill 打包前的基础格式检查

如果没有运行这些验证，要在输出里明确说明当前结论属于静态审查。

### 6. 形成结论

先分开判断两层问题：

- 核心质量是否已足够支撑继续迭代
- 共享与发布准备度是否已经达到当前用户要求

不要把这两层混成一句笼统结论。

## 红旗信号

出现以下想法时，先停下。这通常说明你正在把审查做偏：

| 想法 | 实际要求 |
|---|---|
| “我只看了 `SKILL.md`，但也能判断整包没问题” | 没看到的资产必须明确列为未评估。 |
| “没有 `scripts/`，所以这个 skill 质量差” | 先判断脚本是否是这个 skill 真的需要的。 |
| “README 不够完善，所以 skill 本体一定不合格” | 共享准备度问题不总是核心阻塞。 |
| “有 `evals.json` 就说明可验证性没问题” | 还要检查评测是否和正文规则对齐。 |
| “用户说要重写，那我就不必先 review” | 这个 skill 的职责是先审查，不是直接改写。 |
| “frontmatter 里顺手把工作流都写完，方便触发” | 描述字段主要承担触发条件，不应替代正文。 |

## 输出格式

请严格使用以下结构：

```markdown
## 结论

通过 | 需修改 | 阻塞

## 审查方式

静态审查 | 静态审查 + 轻量验证

## 审查范围

- 已审:
  - <文件或目录>
- 未评估:
  - <文件、目录或维度；若无则写“无”>

## 关键发现

- [critical|important|minor] 问题

## 分维度判断

- 触发与描述: <判断>
- 核心说明质量: <判断>
- 包结构与渐进披露: <判断>
- 评测准备度与对齐性: <判断>
- 一致性与可维护性: <判断>
- 安全与意外性: <判断>
- 共享与发布准备度: <判断；若本轮不关注可写“不作为本轮主判定项”>

## 共享/发布判断

可共享 | 修订后再审 | 输入不足 | 不作为本轮结论
```

如果用户只要求审单个文件，或本轮只是专项审查某一维度，可以在 `审查范围` 中明确说明本次不是 full-package review。

如果用户本轮没有要求共享或发布判断，在 `共享/发布判断` 中写 `不作为本轮结论`。

如果用户明确要求共享或发布判断，但当前输入不足以支撑这类判断，在 `共享/发布判断` 中写 `输入不足`。

## 判定规则

只有在以下条件同时满足时，才返回 `通过`：

- 已审范围内没有 critical 问题
- `description`、正文、评测之间没有明显冲突
- 输出边界清楚，不会把用户带到意料之外的动作
- 如果用户关心共享或发布，当前包至少没有会明显妨碍共享的关键缺口

以下情况返回 `需修改`：

- skill 整体方向可用，但存在重要缺口
- 评测存在对齐问题
- 审查对象可以继续迭代，但离共享或发布还有明确差距

以下情况返回 `阻塞`：

- 缺少 `SKILL.md`
- 关键文件不可读，导致本轮目标无法成立
- 用户要求整包审查，但你几乎拿不到任何包级输入
- 正文存在严重冲突，导致无法判断这个 skill 到底想让代理做什么

## 反模式

- 把审查直接写成重写稿
- 因为想给建议，就跳过正式结论
- 把所有缺口都按同一严重级别处理
- 只盯着行文风格，不判断触发、边界和评测
- 只给“感觉不错/感觉一般”的主观印象，不给可复核的依据
- 没区分“可继续迭代”和“适合共享或发布”

## 完成条件

只有在以下条件满足时，这个 skill 才算完成：

- 已明确本轮审查范围
- 已给出 `通过` / `需修改` / `阻塞` 的唯一结论
- 已给出关键发现和分维度判断
- 若本轮涉及共享或发布，已明确 `可共享` / `修订后再审` / `输入不足` 的共享判断
- 若本轮不涉及共享或发布，已明确写出 `不作为本轮结论`
- 如果未做轻量验证，已显式说明当前结论属于静态审查
