# `skill-reviewer`

## 简介

`skill-reviewer` 是一个独立 skill，用于审查一个 skill 包或单个 `SKILL.md` 的质量。

它关注的不是“直接帮你重写 skill”，而是先判断：

- 这个 skill 会不会在合适场景被触发
- 正文规则是否清晰、自洽、边界稳定
- 包结构是否完整到足以支持理解、复用和维护
- `evals` 是否真的在验证正文，而不是奖励错误行为
- 当前对象是“可以继续迭代”，还是“已经适合共享或发布”

## 目录结构

```text
skill-reviewer/
├── README.md
├── SKILL.md
└── evals/
    ├── README.md
    └── evals.json
```

各文件职责：

- `README.md`：面向使用者的说明文档
- `SKILL.md`：skill 本体
- `evals/README.md`：评测目标与评分关注点
- `evals/evals.json`：评测用 prompts

## 适用场景

以下场景适合使用 `skill-reviewer`：

- 你已经有一个 skill 包，想先做质量审查，而不是直接改写
- 你想知道某个 skill 现在是“可继续迭代”还是“可以共享/发布”
- 你怀疑某个 skill 的 `description` 过宽、过窄，或把正文流程塞进了 frontmatter
- 你想检查 `README`、`evals`、`references`、`scripts` 与 `SKILL.md` 是否一致
- 你担心现有评测 prompt 与正文规则打架，想做一次 package-level review
- 你只拿到单个 `SKILL.md`，希望先做局部质量判断，并明确哪些部分仍未评估

## 不适用场景

以下情况不建议优先使用这个版本：

- 你当前目标是直接创建一个全新的 skill
- 你已经有明确的审查结论和修订清单，只想直接执行改写
- 你关心的是某段代码实现质量，而不是 skill 文档质量
- 你只想让代理顺手润色行文，不关心触发、边界、评测或共享准备度

## 核心能力

`skill-reviewer` 会做 7 件事：

1. 先确认本轮是单文件审查还是整包审查。
2. 读取完成本轮判断所需的最小输入，而不是机械扫完整个仓库。
3. 审查 `description` 的触发质量，而不只看正文写得顺不顺。
4. 判断 `README`、`evals`、`references`、`scripts` 是否与 skill 类型匹配。
5. 检查 `evals` 是否真的在验证正文规则，并暴露“评测奖励错误行为”的问题。
6. 区分“核心质量缺口”和“共享或发布准备度缺口”。
7. 输出结构化结论，但默认不直接改写 skill。

它不会因为缺少可选资产就机械判死刑，也不会在只读到一部分文件时假装已经完成 full-package review。

## 推荐输入

为了让它更高效，建议尽量提供：

- 要审的 skill 目录路径，或至少提供 `SKILL.md`
- 是否要审“整个 skill 包”还是“只审主文件”
- 当前重点是“可继续迭代”还是“可共享/发布”
- 相关的 `README`、`evals`、`references`、`scripts`
- 如果你关心发布准备度，最好说明是否希望它补一轮轻量验证

如果输入不完整，`skill-reviewer` 会显式列出未评估项，而不是假装结论已经覆盖整包。

## 推荐使用方式

最简单的调用方式，是明确告诉它先做审查，不要直接改。

```text
请先用 skill-reviewer 审一下这个 skill 包的质量，判断现在是可共享、修订后再审，还是输入不足。先别重写。
```

如果你只给了主文件：

```text
我现在只给你这个 SKILL.md。先按单文件范围审质量，不要假装已经看过整个 skill 包。
```

如果你更关心触发质量：

```text
请重点看这个 skill 的 description 会不会误触发或漏触发，以及它有没有把正文流程塞进 frontmatter。
```

如果你本来想直接改写，但希望先审再改：

```text
我本来想直接重写这个 skill，但你先用 skill-reviewer 做一次正式审查，确认边界、缺口和共享准备度，再决定要不要进入修订。
```

如果你怀疑评测和正文打架：

```text
请帮我专门审一下这个 skill 的 evals 是否在奖励错误行为，或者有没有漏掉正文里的关键规则。
```

## 典型工作流

### 1. 范围确认

先判断本轮是：

- 单文件审查
- 整包审查
- 聚焦某一维度的专项审查，例如只看 `description` 或只看 `evals`

### 2. 最小输入读取

最少读 `SKILL.md`。

如果要做整包审查，继续读取：

- `README.md`
- `evals/README.md`
- `evals/evals.json`
- 正文明确依赖的 `references` 或 `scripts`

### 3. 类型判断

`skill-reviewer` 会先判断当前对象更像：

- 自包含流程 skill
- 文档型 skill
- 工具型 skill
- 评测驱动型 skill

这样它才能决定哪些资产是必要的，哪些只是可选增强。

### 4. 分维度审查

默认会看这些维度：

- 触发与描述
- 核心说明质量
- 包结构与渐进披露
- 评测准备度与对齐性
- 一致性与可维护性
- 安全与意外性
- 共享与发布准备度

### 5. 轻量验证

如果用户明确关心“可发布”或“可安装”，并且仓库里有现成脚本，它可以补一轮轻量验证。

如果没有跑验证，它会在结果里明确说明“这是静态审查结论”。

### 6. 输出结论

它会给出：

- 唯一结论：`通过` / `需修改` / `阻塞`
- 已审范围和未评估项
- 关键发现
- 分维度判断
- `可共享` / `修订后再审` / `输入不足` / `不作为本轮结论` 的共享判断

## 结论解释

### `通过`

表示在本轮已审范围内，没有足以阻断继续使用的关键问题。

这不自动等于“整包已经可对外分发”，还要结合共享与发布准备度判断。

如果本轮只是单文件审查或专项审查，而且用户本来就没有要求共享或发布判断，`共享/发布判断` 可以是 `不作为本轮结论`。

如果用户明确要求共享或发布判断，但输入只够做局部审查，`共享/发布判断` 应写成 `输入不足`。

### `需修改`

表示整体方向可用，但还存在重要缺口。

最常见的情况包括：

- `description` 与正文范围不一致
- `evals` 在奖励错误行为
- README、SKILL、evals 之间有漂移
- 可以继续迭代，但还不适合共享或发布

### `阻塞`

表示当前输入或 skill 本身存在基础性问题，导致本轮目标无法成立。

最常见的情况包括：

- 缺少 `SKILL.md`
- 关键文件不可读
- 用户要求整包审查，但几乎没有包级输入
- 正文冲突严重，无法判断 skill 的真实边界

## 使用边界

`skill-reviewer` 默认是 audit-only。

这意味着它会：

- 先给正式审查结论
- 明确已审范围和未评估项
- 解释为什么是这个结论

它默认不会：

- 直接把审查结果改写成新版 skill
- 跳过结论直接给一份重写稿
- 因为少了某个可选文件就机械判死刑
- 在未见完整输入时声称已经完成整包验收
