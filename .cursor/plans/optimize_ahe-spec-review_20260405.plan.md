# 优化 `ahe-spec-review` 方案

## 目标

把 `skills/ahe-spec-review/SKILL.md` 从“合格的规格评审 skill”提升为“更稳定判断规格是否可进入真人确认”的门禁 skill。

## 当前短板

- 触发条件主要隐含在 workflow 中，正文里还可以更明确地写“什么状态下该用它”。
- 已有验收标准检查，但缺少更明确的“验收标准质量条”，容易只检查有无，不检查强弱。
- `需修改` 和 `阻塞` 虽然有判定规则，但可以再补一层判断提示，降低边界漂移。

## 优化方向

### 1. 收紧 description 与适用时机

让触发条件更接近“规格草稿已形成、需要判断是否达到真人确认前质量门”的语义。

### 2. 补强验收标准质量条

明确：

- 验收标准应与核心行为对应
- 至少覆盖关键正向和关键负向口径
- 不能只有泛泛成功描述

### 3. 补强 `需修改` / `阻塞` 的区分提示

将边界更明确地写成：

- 仍能继续修补并保留主结构的，倾向 `需修改`
- 核心范围、关键问题或关键矛盾未定的，倾向 `阻塞`

## 明确不做

- 不把 spec review 变成 design review
- 不改变 tri-state 输出格式
- 不改变“通过后仍需真人确认”的契约

## 计划中的实际改动

- 收紧 description
- 增加适用时机
- 增加验收标准质量提示
- 增加结论边界提示

## 预期效果

- 更稳定地评审规格是否已达到可确认状态
- 更少出现“有验收标准但其实很弱”的通过误判
