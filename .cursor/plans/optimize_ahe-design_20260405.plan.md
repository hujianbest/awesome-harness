# 优化 `ahe-design` 方案

## 目标

把 `skills/ahe-design/SKILL.md` 从“能产出设计草稿”提升为“能更稳定地产出可评审实现设计”的 skill。

## 当前短板

- 触发时机主要靠前后文理解，正文里还能更明确。
- 要求比较 2 个方案，但缺少统一的比较维度模板。
- 测试策略虽然有要求，但边界不够明确，容易写成测试细案或漏写。
- 开放问题和待确认风险还可以更显式地作为设计收口的一部分。

## 优化方向

### 1. 收紧 description 与适用时机

更明确写成“规格已批准、需要形成 HOW 设计草稿”的触发语义。

### 2. 增加候选方案比较模板

至少从以下维度比较：

- 工作方式
- 优点
- 缺点
- 对约束 / NFR 的影响
- 为什么适合或不适合当前规格

### 3. 明确测试策略边界

强调设计阶段只定义测试层面的设计准备度、测试策略和风险覆盖思路，不进入任务级测试细案。

### 4. 增加设计开放问题意识

将仍需确认但不阻塞评审的点显式留在设计文档中，避免把设计伪装成完全确定。

## 明确不做

- 不改变“设计通过后仍需真人确认”契约
- 不把设计写成任务计划
- 不把设计写成逐行实现说明

## 计划中的实际改动

- 收紧 description
- 增加适用时机
- 增加候选方案比较模板
- 收紧测试策略边界
- 增加开放问题提示

## 预期效果

- 设计方案比较更一致
- 设计评审时更容易判断是否真的达到任务规划准备度
