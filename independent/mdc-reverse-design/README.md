# `mdc-reverse-design`

## 简介

`mdc-reverse-design` 是一个独立 skill，用于从存量代码仓库中逆向整理出一份 **MDC 风格的详细设计草稿**。

它关注的不是“想象系统本来应该怎么设计”，而是先把当前代码里能确认的架构、模块职责、接口、数据流、约束和测试痕迹整理清楚，再把无法由代码直接证明的部分显式标成 `设计推断` 或 `待确认`。

它只适用于**当前实现的现状补档**，不适用于目标态架构、重构方案或正向设计决策。

各文件职责：

- `README.md`：面向使用者的说明文档
- `SKILL.md`：skill 本体
- `references/reverse-design-template.md`：默认逆向设计章节模板与证据标记约定
- `evals/README.md`：评测目标说明
- `evals/evals.json`：评测用 prompts

补充说明：

- 仓库开发版包含 `evals/` 和 `fixtures/`，方便继续迭代和评测
- 打包后的 `.skill` 分发产物默认不包含 `evals/`，只保留安装和运行所需文件

## 分享与安装

如果你准备把它分享给同事使用，推荐分发**打包后的 `.skill` 文件**，而不是整段仓库源码。

建议做法：

1. 在仓库内先完成静态校验和必要评测。
2. 先确认当前机器上可用的 Python 启动方式，例如 `python3`、`python`、`py -3`，或某个解释器的绝对路径。
3. 在 `.cursor/skills/skill-creator/` 目录下运行仓库约定的模块命令，把 `mdc-reverse-design/` 打包成 `.skill` 文件：

```bash
python3 -m scripts.package_skill ../../../independent/mdc-reverse-design ../../../independent/dist
```

如果当前环境没有 `python3` 这个命令别名，请把上面的 `python3` 替换成你本机可用的等价启动方式。

4. 让输出目录位于 skill 源码目录之外，避免把 `.skill` 产物再次打包进自己。
5. 对外分享时，默认分享 `.skill` 产物；`evals/` 和 `fixtures/` 更适合留在源码仓库里继续维护。

这意味着：

- 使用者安装时主要依赖 `SKILL.md`、`README.md` 和 `references/`
- `evals/` 是作者侧资产，不是安装后运行 skill 的必需输入

## 适用场景

以下场景适合使用 `mdc-reverse-design`：

- 老项目或 brownfield 仓库缺少可信的设计文档，需要先补一份现状设计草稿
- 想从代码出发梳理当前架构，而不是直接做目标态设计
- 只想先分析某个模块、某条主流程或某个子系统，而不是一次写完整仓设计
- 代码、注释和旧文档不完全一致，需要先把“代码事实上如何工作”整理出来
- 已经有一份旧设计草稿，但需要结合当前代码继续补全，而不是整份重写
- 在正式设计评审前，需要一份事实优先的 MDC 风格草稿作为输入

## 不适用场景

以下情况不建议优先使用这个版本：

- 只有业务想法，没有可分析的代码仓库
- 当前目标是澄清需求、产品范围或非目标，而不是梳理现状代码
- 当前目标是产出目标态架构方案、未来改造方案或正式设计决策
- 当前重点已经是任务拆解、代码实现、调试或 hotfix

如果用户真正需要的是需求澄清或正向设计，`mdc-reverse-design` 不应该假装自己能够替代这些阶段。

## 核心能力

`mdc-reverse-design` 会做 6 件事：

1. 读取逆向补档所需的最小上下文，而不是盲目扫完整仓。
2. 在输入不足或范围过大时，先停下来补齐边界，而不是假装能直接补全设计。
3. 从代码中提取入口、模块边界、接口、数据流、依赖和测试痕迹。
4. 把结论明确区分为 `代码事实`、`设计推断` 和 `待确认`。
5. 生成一份 MDC 风格的详细设计草稿，而不是文件遍历或实现讲解。
6. 在结束前做一轮轻量自检，并给出当前 readiness 判断。

它不会替当前实现背书，也不会把代码里看不出来的业务意图写成既定事实。

## 推荐输入

为了让它更高效，建议尽量提供：

- 仓库路径或明确的模块范围
- 入口文件、启动文件、主链路或关键目录
- 已知的技术栈、运行方式或部署线索
- 旧设计稿、README、注释说明或历史背景材料
- 指定的文档路径、模板或章节要求
- 你最关心的补档范围，例如“只看支付子系统”或“只梳理订单提交流程”

如果输入不足，`mdc-reverse-design` 会先指出阻塞项，而不是强行产出一份“看起来很完整”的设计文档。

## 推荐使用方式

最简单的调用方式，是明确告诉它要从代码逆向整理现状设计，而不是做目标态方案。

```text
这个仓库是个老项目，没有设计文档。请先按事实优先方式梳理当前架构，再生成一份 MDC 风格的详细设计草稿。
```

如果你只想分析一个子系统：

```text
这是一个 monorepo。请只分析通知子系统当前的代码结构和主流程，补一份 MDC 风格设计草稿，不要把支付和报表模块一起展开。
```

如果你想先盘点事实，再落设计草稿：

```text
请先区分代码事实、设计推断和待确认项，再据此整理成一份 MDC 风格的设计文档草稿。
```

如果你希望它按特定路径和模板落盘：

```text
请基于现有代码梳理订单审批模块的现状设计，文档写到 docs/designs/order-approval-reverse-design.md，并使用 概述 / 架构设计 / 模块职责 / 数据流与控制流 / 接口与契约 / 测试策略 / 风险与待定问题 这套章节。
```

如果你已经有旧草稿，想在现有基础上补完：

```text
这是我现有的逆向设计草稿，请先保留现有结构，只根据当前代码补齐缺口并标出哪些结论是代码事实、哪些还是待确认。
```

## 典型工作流

### 1. 初始输入

你先提供仓库、模块或主链路范围，以及你想补的设计层级。

### 2. 输入体检

`mdc-reverse-design` 会先检查：

- 是否存在可读的代码范围
- 最小入口或主模块是否明确
- 当前目标是不是现状逆向补档
- 范围是否过大，需要先切边界

### 3. 范围切分

如果请求明显过大，它不会直接写整仓总设计，而会先给出 2-3 种合理切法，例如：

- 按服务或部署单元拆
- 按业务域或子系统拆
- 按主运行时链路拆

然后推荐本轮先补哪一份设计草稿。

### 4. 提取代码事实

在范围明确后，它会优先从以下线索整理设计：

- 启动入口和装配方式
- 模块边界与依赖方向
- 对外接口与内部契约
- 数据模型、存储与外部依赖
- 日志、鉴权、校验和错误处理
- 测试目录、测试工具和 CI 命令

### 5. 起草设计文档

它会使用 MDC 风格章节组织结果，默认路径是：

- `docs/designs/YYYY-MM-DD-<topic>-reverse-design.md`

如果没有现成模板，默认参考：

- `references/reverse-design-template.md`

### 6. 轻量自检

文档起草后，它会检查：

- 关键结论是否都带有证据边界
- 是否明确覆盖模块职责、接口、数据流和测试策略
- 是否把冲突和未知项显式写出来
- 是否误把实现细节、改造建议或任务拆解写进设计文档

最后只会给出两种判断之一：

- `可提交设计评审`
- `仍需补充代码探索或业务上下文`

## 输出内容

完成时通常会得到两类结果：

### 1. 设计文档草稿

这是主要产物，会落盘到你指定的路径，或默认路径：

- `docs/designs/YYYY-MM-DD-<topic>-reverse-design.md`

文档默认是基于代码事实逆向整理的 `状态: 草稿`，不是已批准设计。

### 2. 收尾判断

结尾会包含简短总结，例如：

```markdown
代码逆向设计草稿已完成。

- 文档路径: `docs/designs/2026-04-01-order-approval-reverse-design.md`
- 当前判断: `仍需补充代码探索或业务上下文`
- 仍待确认:
  - 通知重试策略是否只存在于线上配置
  - 批量审批链路是否由异步 worker 处理
```

如果输入还不够，它会明确停在“当前还不能可靠地完成代码逆向设计补档”，而不是伪装成已经有了可评审文档。

## 常见误区

使用 `mdc-reverse-design` 时，常见问题包括：

- 没有给仓库范围，却希望一次补完整仓设计
- 把“当前实现存在”误当成“原始设计就是这样”
- 明明需要目标态设计，却让它硬从代码倒推出未来方案
- 代码里看不出来的内容也不愿保留 `待确认`
- 把设计草稿写成文件遍历、类图讲解或任务拆解

如果出现这些情况，应该先缩小范围、补充上下文或切回正确阶段，而不是继续硬写。

## 总结

如果你的目标是“先把存量代码当前的结构、边界和主链路讲清楚，再补一份能进入评审的设计草稿”，`mdc-reverse-design` 就是合适入口。

它最适合老系统补档、现状架构梳理、模块级 reverse design，以及需要把代码事实和设计推断严格分开的场景。
