# 逆向设计模板

当用户没有指定设计模板时，使用本文件作为 `mdc-reverse-design` 的默认章节骨架。

目标不是把代码复述成实现笔记，而是把“当前代码如何工作”整理成一份可评审的 MDC 风格设计草稿。

## 证据标记约定

默认在各章节内使用以下标签：

| 标签 | 含义 | 使用要求 |
|---|---|---|
| `代码事实` | 能被代码、配置、测试或明确工件直接支持的结论 | 尽量给出来源范围，如入口、模块、契约或配置位置 |
| `设计推断` | 基于多个代码事实归纳出的设计解释 | 必须显式说明这是推断 |
| `待确认` | 代码无法证明、输入互相冲突、或依赖业务上下文才能判断的内容 | 说明它为什么不确定，以及影响范围 |

如果某段内容只来自注释、命名习惯或历史猜测，不要把它写成 `代码事实`。

## 默认文档模板

```markdown
# <主题> 现状逆向设计草稿

- 状态: 草稿
- 来源: 基于代码事实逆向整理
- 分析范围: <仓库 / 模块 / 服务 / 子系统>
- 主要证据: <入口、目录、配置、测试或其它关键工件>

## 概述

- `代码事实:` <当前范围内系统或模块的直接观察结果>
- `设计推断:` <基于代码结构归纳出的系统定位或组织方式>
- `待确认:` <仍缺少的业务背景或运行信息>

## 设计驱动因素

- `代码事实:` <现有代码明确体现的约束、依赖、部署假设、兼容性条件>
- `设计推断:` <这些因素如何影响当前结构划分>
- `待确认:` <性能、安全、合规、容量等仍未能由代码证明的约束>

## 表达视角比较

只有当当前代码可以被多种方式合理表达时，才比较 2 到 3 种视角：

### 视角 A: <例如按分层结构表达>
- 适用原因:
- 优点:
- 局限:

### 视角 B: <例如按业务域表达>
- 适用原因:
- 优点:
- 局限:

### 视角 C: <可选，例如按主运行链路表达>
- 适用原因:
- 优点:
- 局限:

## 推荐表达视角

- `代码事实:` <为什么当前代码更适合用该视角解释>
- `设计推断:` <该表达方式如何减少后续评审中的猜测空间>
- `待确认:` <还有哪些模块或链路可能需要另一种视角补充>

## 架构设计

- `代码事实:` <主要运行单元、边界、入口、装配方式>
- `设计推断:` <当前整体架构形态，例如分层、模块化单体、事件驱动等>
- `待确认:` <静态阅读无法确认的运行时拓扑、部署或调度关系>

## 模块职责

### <模块或子系统 A>
- `代码事实:` <它包含什么、暴露什么、依赖什么>
- `设计推断:` <它在整体中的职责边界>
- `待确认:` <边界是否与业务概念完全一致>

### <模块或子系统 B>
- `代码事实:`
- `设计推断:`
- `待确认:`

## 数据流与控制流

### <主流程或链路 1>
- `代码事实:` <入口 -> 校验 -> 业务逻辑 -> 存储 / 外部调用 -> 返回>
- `设计推断:` <该链路体现出的职责协作关系>
- `待确认:` <异步边、补偿、重试或外部触发条件中无法确认的部分>

### <主流程或链路 2>
- `代码事实:`
- `设计推断:`
- `待确认:`

## 接口与契约

- `代码事实:` <HTTP / RPC / CLI / 事件 / 导出接口 / schema / config contract>
- `设计推断:` <哪些契约构成稳定边界，哪些更像内部实现细节>
- `待确认:` <调用方约束、版本策略或兼容性规则里未能确认的内容>

## 测试策略

- `代码事实:` <测试目录、框架、CI 命令、已覆盖链路>
- `设计推断:` <当前测试更偏单元、集成、端到端还是冒烟>
- `待确认:` <缺失的验证层、运行环境假设、未覆盖的高风险区域>

## 风险与待定问题

- `代码事实:` <代码异味、结构冲突、重复实现、注释与行为不一致等>
- `设计推断:` <这些问题可能对后续评审、改造或任务拆解造成的影响>
- `待确认:` <必须补充业务背景、运行信息或人工确认的问题>
```

## 代码证据到设计章节的映射

| 代码证据 | 优先写入的章节 | 说明 |
|---|---|---|
| 启动文件、路由注册、主命令入口 | `概述` / `架构设计` / `数据流与控制流` | 用来确认系统从哪里开始运行 |
| 目录结构、模块边界、依赖方向 | `架构设计` / `模块职责` | 用来确定边界和职责分层 |
| API 路由、公共导出、消息主题、schema | `接口与契约` | 用来识别对外或跨模块边界 |
| 实体、迁移、仓储、缓存、队列 | `数据流与控制流` / `设计驱动因素` | 用来识别状态流转和持久化依赖 |
| 配置项、环境变量、第三方客户端 | `设计驱动因素` / `风险与待定问题` | 用来识别环境依赖与外部约束 |
| 测试目录、测试框架、CI 命令 | `测试策略` | 用来描述现有验证方式，而不是想象中的理想策略 |
| 注释、README、旧文档 | 仅作辅助输入 | 与代码冲突时，不得压过代码事实 |

## 低置信度写作规则

- 只读到单个模块或单条链路时，不要做“整仓架构”结论。
- 如果已有文档、注释和代码彼此冲突，显式写进 `待确认` 或 `风险与待定问题`。
- 静态阅读无法确认的运行时行为，例如真实部署拓扑、灰度策略、补偿机制、外部系统约束，不要写成确定事实。
- “当前实现存在”不等于“这是合理或被确认的设计”；需要把现状和评价分开。
- 当用户施压要求“直接补全完整设计”时，仍应优先保留证据边界，而不是用自信语气掩盖未知项。

## 不应产出的内容

- 逐文件摘要
- 类 / 函数级实现讲解
- 伪代码
- 任务拆解
- 未经证据支持的业务目标或产品策略
