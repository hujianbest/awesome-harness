# MDC 代码逆向设计补档

基于现有代码仓库，整理一份可供继续补全和评审的 MDC 风格设计草稿。

核心原则：先记录代码事实，再做最小必要推断；拿不准的地方明确标成 `待确认`，不要把当前实现粉饰成已批准设计。

## 硬性门禁

在设计草稿足够可靠之前，不要：

- 声称这是一份已批准设计
- 把代码里看不出来的业务意图写成既定事实
- 用文件逐行解说、伪代码或任务清单替代设计表达
- 在缺少仓库范围、入口或关键上下文时假装已完成补档

如果用户只有业务想法、目标状态需求或改造方案，而不是要梳理当前代码现状：

- 不要把“代码逆向补档”伪装成需求澄清或正向设计
- 这类场景应回到需求或设计阶段处理，而不是继续硬写“as-built 设计”

如果项目采用完整 `mdc-workflow`：

- 这份文档仍只是 `状态: 草稿` 的逆向整理结果
- 只有经过正式设计评审与确认后，才算进入正式设计工件

## 目标

最终草稿至少要回答这些问题：

- 当前范围内主要模块或组件是什么
- 模块边界和职责如何划分
- 主要数据流与控制流如何运行
- 对外接口、内部契约和外部依赖是什么
- 有哪些约束、交叉关注点和测试痕迹
- 哪些结论来自代码事实，哪些只是设计推断
- 还有哪些缺口会阻塞后续设计评审或补全

## 快速工作模式

| 情况 | 当前回合应做什么 | 不要做 |
|---|---|---|
| 用户未给仓库路径、入口或分析范围 | 先索取最小必要输入，只问会影响补档的关键问题 | 不要假装已经能逆向出整仓设计 |
| 仓库明显过大，包含多个服务或子系统 | 先给 2-3 种切分方式并推荐本轮范围 | 不要直接写一份覆盖全仓的失真总设计 |
| 代码结构较清晰，用户只想补当前设计草稿 | 直接抽取代码事实并起草文档 | 不要把文档写成代码导读或实现笔记 |
| 用户指定了路径、模板或章节骨架 | 严格按用户给定要求落盘 | 不要偷偷退回默认路径或模板 |
| 用户要求“尽量补全，不要总写待确认” | 保持事实优先，只在有证据时下结论 | 不要为“看起来完整”而过度断言 |
| 代码、注释、已有文档彼此冲突 | 显式暴露冲突、影响和待确认项 | 不要静默选边并继续写成定论 |

## 工作方式

### 1. 先读取最小必要上下文

优先读取：

- 用户指定的仓库范围、模块范围或入口文件
- `AGENTS.md` 中与设计路径、模板、团队规范相关的约定
- 项目 `README`、启动入口、路由注册、服务装配、配置和测试入口
- 用户明确给出的旧文档、章节骨架或落盘路径

如果用户提供旧设计稿或注释性文档，把它视为参考输入，不要直接当成正确答案。

### 1.1 修订旧草稿时，先 intake 再补稿

如果用户说“在现有设计草稿上补完”“不要整份重写”：

- 先拿到旧草稿正文或明确路径，再开始补稿
- 在未看到旧草稿前，只能做 intake/triage：说明还缺什么，不能假装已经完成修订
- 真正补稿时，优先保留原有章节骨架、已确认事实和未受影响的内容
- 只补与当前代码事实直接相关的缺口，并把新增或修正内容标成 `代码事实`、`设计推断` 或 `待确认`
- 如果必须替换原结论，要说明是因为代码证据不足、旧文档与现状冲突，还是当前范围已经变化

### 2. 先判断是否具备逆向补档条件

先确认当前输入是否已经足以支撑“有边界、有职责、有流转、有证据”的设计草稿。

以下情况通常说明现在还不能可靠补档：

- 没有可读的代码范围或仓库路径
- 连最小入口、主模块或分析目标都不明确
- 用户实际要的是目标态设计，而不是现状逆向整理
- 仓库过大，但当前回合尚未选定补档边界

此时应：

- 回放已确认事实
- 明确列出缺失输入和阻塞项
- 只追问一个当前最关键的问题

不要在这些阻塞项存在时硬写一份“看起来完整”的设计草稿。

### 3. 必要时先切分补档范围

当仓库明显过大时，先给 2-3 种合理的补档切法，再推荐一个：

- 按部署单元或服务拆
- 按业务域或子系统拆
- 按主要运行时链路拆

说明每种切法的取舍，然后优先选出本轮最值得补的一份设计草稿。

### 4. 提取代码事实

只读完成当前补档所必需的最小上下文，优先提取：

- 入口与启动链路
- 目录、包、服务和模块边界
- 对外接口、内部公开契约、消息或事件边
- 数据模型、存储、缓存、队列与配置依赖
- 错误处理、鉴权、校验、日志等交叉关注点
- 测试目录、测试工具、CI/脚本里能看到的验证方式

重点是整理“当前代码事实上如何工作”，而不是逐文件复述实现细节。

### 5. 区分事实、推断与未知

写草稿时默认使用以下标签：

| 标签 | 何时使用 | 写法要求 |
|---|---|---|
| `代码事实` | 可被代码、配置、测试或明确工件直接支持 | 尽量指出来源范围，如入口、目录、模块或契约位置 |
| `设计推断` | 由多个代码事实归纳出的较稳定设计解释 | 说明它是推断，不要伪装成确定事实 |
| `待确认` | 代码看不出来、存在冲突、或依赖业务上下文才能判断 | 说明为什么无法确认，以及它会影响哪一部分设计 |

如果某个结论只能由“命名习惯”或“感觉上应该如此”支撑，不要把它写成 `代码事实`。

### 6. 必要时比较 2 到 3 种表达视角

如果当前代码可以被多种合理方式解释，不要直接选一个就当成唯一设计。

至少比较两种表达方式，例如：

- 以分层结构为主线
- 以业务域为主线
- 以主运行链路为主线

最后推荐最能解释当前代码结构的一种，并说明为什么。

注意：这里比较的是“如何表达当前设计”，不是借机改写成未来目标态方案，也不是要求凭空发明多个未来方案。

### 7. 编写设计文档

若用户或项目已指定路径、命名方式或模板，优先遵循。

若没有指定，默认保存到：

- `docs/designs/YYYY-MM-DD-<topic>-reverse-design.md`

若没有现成模板，使用 `references/reverse-design-template.md` 作为默认章节骨架。

编写要求：

- 文档顶部至少体现 `状态: 草稿`
- 让标题或路径能看出这是一份“现状 / 逆向”设计草稿，而不是已定稿的正向设计
- 明确写出当前分析范围和主要证据基础
- 保持在设计层抽象，不写成伪代码、逐文件导读或任务清单
- 对关键模块、数据/控制流、接口与契约、测试策略给出足够支撑后续评审的说明
- 不拿“当前实现存在”来替代“设计合理”
- 当信息不足时，宁可保留 `待确认`，也不要臆造完整意图

## 轻量自检

起草完成后，至少检查：

- 主要结论是否都能落到 `代码事实`、`设计推断` 或 `待确认`
- 是否明确覆盖了模块边界、职责、接口、数据流、约束和测试策略
- 是否把大仓库范围切小，而不是失真地写成全景总设计
- 是否遵循了用户指定的路径、模板或章节骨架
- 是否把冲突、缺口和低置信度区域显式写出
- 是否混入了逐行实现讲解、伪代码或任务拆解
- 如果用户要的是目标态设计而不是现状补档，是否已经及时停止并指出不匹配

如果仍存在会影响评审结论的关键缺口，不要假装已经可以提交设计评审。

## 输出格式

不是每一轮都应该输出“代码逆向设计草稿已完成”。

### A. 当前还不能可靠地完成代码逆向设计补档

如果缺少代码范围、入口、关键上下文，或当前请求更像需求/目标态设计，使用：

```markdown
当前还不能可靠地完成代码逆向设计补档。

- 已确认:
  - <已知事实>
- 缺失输入:
  - <缺少的代码范围、入口、路径或上下文>
- 仍待确认:
  - <阻塞项>
- 下一步:
  - <只保留一个当前最关键的补齐动作>
```

此模式下：

- 不要假装设计草稿已完成
- 不要输出 `可提交设计评审`
- 不要用大段模板文字掩盖输入不足

### B. 代码逆向设计草稿已形成

只有在以下条件同时满足时，才使用 completion block：

- 本轮已实际起草、补全或实质更新了一份设计草稿
- 文档路径已明确
- 事实、推断和待确认已区分清楚
- 已完成一轮轻量自检

完成时输出应包含：

```markdown
代码逆向设计草稿已完成。

- 文档路径: `<design-path>`
- 当前判断: `可提交设计评审` | `仍需补充代码探索或业务上下文`
- 仍待确认:
  - <若无则写“无”>

如需，我可以继续补充证据、缩小范围，或根据新增业务上下文修订这份设计草稿。
```

只有当主要边界、职责、接口、数据流和关键约束已经有足够证据支撑，且剩余问题不再阻塞评审时，才能判断为 `可提交设计评审`。

## 反模式

- 把当前实现合理化成“原始设计意图”
- 代码里看不出来的业务目标却被写成既定事实
- 把设计文档写成文件遍历、伪代码或类/函数讲解
- 仓库范围明显过大，却不切边界直接写总设计
- 为了让文档看起来完整，故意隐藏冲突和 `待确认`
- 顺手把任务拆解、改造建议或实现方案写进当前草稿
- 用户明明指定了路径或模板，却退回默认输出

## 完成条件

如果当前回合仍处在“补齐输入”“选定范围”或“暴露冲突”的阶段，正确停在“当前还不能可靠地完成代码逆向设计补档”并等待更多输入，是允许的中间状态。

只有在以下条件满足时，这个 skill 才算完成：

- 当前分析范围已经明确
- 设计草稿已实际落盘
- 主要结论与不确定项已被正确标记
- 已完成一轮轻量自检
- 已明确给出 `可提交设计评审` 或 `仍需补充代码探索或业务上下文` 的判断
