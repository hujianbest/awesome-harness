# `mdc-specify`

## 简介

`mdc-specify` 是一个独立 skill，用于把模糊想法逐步澄清成一份可用于后续设计的规范文档。

它适合单独使用，不依赖 `mdc-workflow-starter`、`mdc-spec-review` 或完整的 `mdc-*` workflow 门禁。

如果你当前要走完整的软件交付流程，请改用 workflow 版 `mdc-specify`；如果你只是想先把需求说清楚、落成 spec，再决定要不要进入设计，这个独立版就是合适入口。

## 目录结构

```text
mdc-specify/
├── README.md
├── SKILL.md
└── evals/
    ├── README.md
    └── evals.json
```

各文件职责：

- `README.md`：面向使用者的说明文档
- `SKILL.md`：skill 本体
- `evals/README.md`：评测目标说明
- `evals/evals.json`：评测用 prompts

## 适用场景

以下场景适合使用独立版 `mdc-specify`：

- 只有一个初步想法，想先把需求澄清清楚
- 想先写 spec，再进入设计
- 需求范围还不稳定，需要先明确目标、用户、范围、非目标和验收标准
- 已经有一份需求草稿，但还不完整，需要继续补齐
- 用户描述里混入了技术细节，需要先抽离出真正的业务需求

## 不适用场景

以下情况不建议优先使用这个版本：

- 当前会话已经处在 `mdc-workflow` 正式链路中
- 你需要 `mdc-spec-review`、真人确认或其它 workflow 门禁
- 已经有了可用且已确认的规格，当前目标是进入设计
- 当前重点是架构设计、任务拆解或代码实现

## 核心能力

`mdc-specify` 会做 5 件事：

1. 读取当前需求相关的最小上下文。
2. 通过一问一答方式逐步澄清需求。
3. 当需求过大或边界不清时，给出 2-3 种范围切分方案。
4. 起草规范文档，而不是提前给设计方案。
5. 在结束前做一轮轻量自检，并给出 readiness 判断。

它不会直接跳到实现，也不会把接口、表结构、类设计、框架选择等实现细节写进需求规范。

## 推荐输入

为了让它更高效，建议尽量提供：

- 你要解决的问题
- 谁会使用这个能力
- 已知范围和限制
- 明确不做的内容
- 成功标准或验收标准
- 现有草稿、旧文档或相关背景材料
- 指定的文档路径、模板或章节要求

信息不全也没关系，`mdc-specify` 会先通过提问帮助补齐。

## 推荐使用方式

最简单的调用方式，是明确告诉它先做需求澄清，不要设计。

```text
我们想做一个内部审批系统，先不要做技术方案，先帮我把需求澄清清楚并整理成 spec。
```

如果你已经有草稿：

```text
这是我现有的需求草稿，请不要当成定稿，先帮我继续澄清缺失项，再整理成规范文档。
```

如果你希望它按特定路径和模板落盘：

```text
规范文档请写到 docs/prd/release-approval-prd.md，并使用 背景 / 目标 / 范围 / 非目标 / 验收标准 / 开放问题 这套章节。
```

## 典型工作流

### 1. 初始输入

你先提供一个需求想法、目标或草稿。

### 2. 需求澄清

`mdc-specify` 会按主题逐步提问，优先澄清：

- 目标与用户
- 功能范围
- 非目标
- 约束与依赖
- 验收标准
- 负路径和边界情况

### 3. 范围切分

如果需求过大，`mdc-specify` 不会把所有内容硬塞进一份 spec，而是先给出 2-3 种合理切分方式，例如：

- 最小可落地范围
- 业务完整范围
- 风险优先范围

然后推荐本轮先写哪一份规范。

### 4. 理解回放

正式起草前，它会先回放当前理解，确认以下内容是否稳定：

- 目标
- 用户
- 范围
- 非目标
- 关键约束
- 成功标准

### 5. 起草规范文档

若你已指定路径或模板，就优先按你的要求生成。

如果没有指定，默认路径是：

- `docs/specs/YYYY-MM-DD-<topic>-spec.md`

如果没有现成模板，默认最小结构包括：

- 背景与目标
- 用户与使用场景
- 范围
- 非目标
- 功能要求
- 非功能要求
- 约束与依赖
- 验收标准
- 边界情况与负路径
- 开放问题

### 6. 轻量自检

文档起草后，它会检查：

- 目标是否清楚
- 范围和非目标是否明确
- 验收标准是否可验证
- 是否混入设计决策
- 是否还有阻塞后续设计的开放问题

最后只会给出两种判断之一：

- `可进入设计`
- `仍需继续澄清`

## 输出内容

完成时通常会得到两类结果：

### 1. 规范文档草稿

这是主要产物，会落盘到你指定的路径，或默认路径。

### 2. 收尾判断

结尾会包含简短总结，例如：

```markdown
需求规范草稿已完成。

- 文档路径: `docs/specs/2026-03-31-release-approval-spec.md`
- 当前判断: `仍需继续澄清`
- 仍待确认:
  - 审批超时是否需要自动提醒
  - 被拒绝后的再次提交规则是否属于一期范围
```

## 常见误区

使用 `mdc-specify` 时，常见问题包括：

- 把想法记录误当成已经可以进入设计的规范
- 只写要做什么，不写明确不做什么
- 把 `/api/...`、数据库表、类、消息队列、重试策略等技术实现混进需求
- 没有写验收标准，只留下笼统表述
- 需求明显过大，但仍想一次写成一份总规范

如果出现这些情况，应该回到澄清阶段，而不是直接进入设计。

## 与 workflow 版的区别

| 维度 | 独立版 `independent/mdc-specify` | workflow 版 `mdc-workflow/skills/mdc-specify` |
|---|---|---|
| 使用方式 | 单独使用 | 作为 `mdc-*` 流程中的一个节点 |
| 前置依赖 | 无 | 依赖 `mdc-workflow-starter` 路由 |
| 收尾方式 | 轻量自检 + readiness 判断 | 交给 `mdc-spec-review` |
| 门禁强度 | 轻量 | 强 workflow 门禁 |
| 适用场景 | 独立需求澄清与 spec 起草 | 完整软件交付流程中的规格阶段 |

## 总结

如果你的目标是“先把需求说清楚，再进入设计”，独立版 `mdc-specify` 就是合适入口。

它最适合需求尚不稳定、范围需要澄清、想先获得一份规范文档草稿的场景；它不替代完整 workflow 门禁，而是帮助你在独立使用时，把需求阶段做扎实。
